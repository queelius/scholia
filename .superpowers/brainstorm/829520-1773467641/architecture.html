Architecture Approach

How should the pieces fit together? Three approaches to the same vision.

A

Evolutionary — Grow What's There

Keep the current architecture (aiohttp + in-memory state). Add CodeMirror extensions for autocomplete. Add SQLite for compile history. Enrich the existing MCP tools with richer context. Add a WebSocket "user focus" channel that reports cursor/selection/scroll to the MCP layer.

Pros

  • Smallest diff — builds on working code
  • No migration pain
  • Can ship incrementally (editor first, then history, then MCP)

Cons

  • server.py is already 1800 lines — grows further
  • MCP ↔ HTTP ↔ WebSocket layering gets complex
  • No clean boundary between "editor features" and "viewer features"
B

Modular Refactor — Split Then Grow

First, decompose server.py into focused modules: editor routes, viewer routes, MCP bridge, compilation engine, persistence layer. Then add features into clean slots. SQLite persistence as a proper layer. MCP context channel as a dedicated WebSocket listener. CodeMirror extensions as a separate JS module.

Pros

  • Each new feature has a clear home
  • MCP bridge becomes a first-class module with its own tests
  • Easier to reason about — each module fits in context
  • SQLite layer is clean (not bolted onto in-memory state)

Cons

  • Upfront refactoring cost before new features
  • Risk of breaking existing functionality during decomposition
  • ~2-3 sessions of refactoring before visible user-facing progress
C

Hybrid — Feature-Driven Decomposition

Don't refactor upfront. Instead, as each feature is built (editor, history, MCP context), extract the relevant code into its own module as part of that feature work. First feature (editor autocomplete) extracts editor concerns. Second feature (SQLite) creates the persistence layer. Third feature (MCP context) creates the awareness module. Same end state as B, but the refactoring pays for itself immediately.

Pros

  • Every session delivers user-facing value
  • Refactoring is justified by the feature it enables
  • Same clean end state as B
  • Lower risk — each extraction is small and tested

Cons

  • Intermediate states may be messy (half-extracted)
  • Requires discipline to actually extract vs just adding

My recommendation: C (Hybrid). You get user-facing progress every session, and the architecture improves as a side effect of building features rather than as an upfront tax. The key risk (not actually extracting) is mitigated by the fact that each feature needs the extraction to be implementable cleanly.
